Closures in JavaScript

Jul 14th 2015

This is my first blog post ever! So what better way to kick this thing off by starting with the simple concept of "Closures"!

(jk..Calm Down.)

While closures are a tricky subject to master at first, please note that it is not impossible. Let me start by giving the textbook definition of a closure and we'll go from there.

Here is the definition of a closure on Mozilla's Developer network:

"Closures are functions that refer to independent (free) variables. In other words, the function defined in the closure 'remembers' the environment in which it was created."

Simple enough right???

Yeah, if only it were that simple.. lets try to make some sense of this.

Hopefully you are familiar with functions in javascript. (if not, please familiarize yourself with functions, as they are used for everything in JavaScript) An easy way to think of a closure is a function that is "RETURNED" inside of another function. That's it! So anytime you see another function inside of a function, just know that the inner function is just a word that some guy made up years ago and decided to call it a "closure". No Big Deal.

What makes a "Closure" so special (and maybe in some ways confusing) is that the inner function or "closure" has access to the outer (parent) function's scope (in other words the variables and anything else defined in the outer (parent) function, even when that parent function ends. Crazy right? So at a very high level without going into the in's and outs of why thats even possible, just know that a closure is a "RETURNED" function inside of another function that has access to its parent functions scope (the variables, etc..), even when that parent function ends. Lets take a look at an example.

say we have defined a function.. lets call it banana... var banana = function(x){ var count = x; return bananaSplit = function(){ console.log("you have", count, "bananas"); } } var nate = banana(23); nate();

the Main takeaways to note are that: 1. the inner function is the closure. 2. bananaSplit has access to the variable count in the parent function. 3. I'm calling the banana function and setting it equal to var nate. the way it works is: banana(23) will invoke the banana function and pass in 23 to the count variable. then it will return the bananaSplit function.. so you can think of it now as: var nate = bananaSplit function. 4. Now when i call nate() it will run or invoke the bananaSplit function which console logs how many bananas we have. and remember bananaSplit still has access to all the variables inside of the banana parent function.

Simple enough right???
